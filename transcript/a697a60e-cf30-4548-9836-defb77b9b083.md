### Why does CI start a Postgres container?

**Yes — it's there so tests run against a real database**, same as local dev.

This project is not SQLite-friendly in CI:

1. Settings default to PostgreSQL.
2. Migrations use Postgres-only features (`pg_trgm`, GIN indexes in `0002_post_performance_indexes`).
3. Tests hit the DB — pytest-django creates a test DB, runs migrations, exercises models/APIs.

Without Postgres in CI, `migrate` and several tests would fail.

**Does it use more resources?** A bit, but it's normal and usually cheap:

- GitHub runs a **service container** next to the job (not a separate billable VM).
- It starts with the job and is torn down after (~1–2 minutes).
- Cost is mostly **job minutes** on the runner.

**Alternatives** (not used — weaker fidelity):

| Approach | Tradeoff |
| -------- | -------- |
| SQLite in CI | Breaks `pg_trgm` migrations |
| Mock DB / no DB | Doesn't test migrations or integration |
| Skip DB tests | Fast but misses regressions |

You could split “lint only” and “test with Postgres” into two jobs; the test job still needs Postgres.
